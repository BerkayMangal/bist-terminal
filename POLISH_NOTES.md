## 1. Limitation #8 — `datetime.utcnow()` deprecation (housekeeping)

**Before:** Three callsites (`engine/bullalfa.py:884`, `engine/bullalfa.py:1335`, `api/bullalfa.py:277`) used `datetime.utcnow()`, deprecated in Python 3.12+. Test output emitted ~270 `DeprecationWarning` per run.

**After:** All three replaced with `datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ")`. Output format is byte-identical — the orchestrator's `generated_at` field still produces e.g. `"2026-05-10T13:42:18Z"`. No schema impact, no test changes required.

**Verification:** Suite re-run, 0 warnings.

## 2. Limitation #6 — Pydantic models + JSON Schema artifact

**Before:** §19 schema enforced inline by `TestSchemaInvariants`, derivable only by reading the orchestrator's return statement. No portable artifact for client-side validation.

**After:**

| Path | Purpose | LOC |
|------|---------|----:|
| `api/bullalfa_schemas.py` | Pydantic v2 models for the §19 surface: `BullAlfaSignal`, `ScanResponse`, `TickerResponse` + 12 sub-block models. Literal types for the constrained enums (mode, grade, regime, phase, sector_group, kalite, value, breakout_type, lifecycle status/outcome). `extra="allow"` config so future schema additions don't immediately reject v1.4-shaped payloads. | 195 |
| `bullalfa_jsonschema.json` | Standalone JSON Schema artifact (Draft 2020-12). 29KB. Generated by `api.bullalfa_schemas.export_json_schema()`. Distributable to API consumers for client-side validation. | (artifact) |
| `tests/test_bullalfa_schemas.py` | 13 tests. Validates orchestrator output across all 6 modes + 4 degradation paths (macro_unavailable, aggregation_failed, pit_missing, short_history). Plus 3 schema-artifact integrity tests. | 222 |

**Design choice:** Pydantic models are NOT enforced as `response_model=` on the route handlers — that would risk strict validation breaking existing tests on edge-case shapes. Instead they're documentation + validation tools. v1.4.1 housekeeping can flip them on once we've shipped a few bars of production traffic without contract surprises.

**Verification:**

```python
>>> from api.bullalfa_schemas import BullAlfaSignal
>>> sig = build_bullalfa_signal(...)  # any of the 6 mode paths
>>> BullAlfaSignal.model_validate(sig)  # never raises in v1.4
```

The `extra="allow"` permissiveness means a future v1.5 that adds a new top-level key (e.g. `lifecycle_v2`) won't break v1.4 consumers — they'll just see the extra key and can choose to ignore it or upgrade.

## 3. Limitation #4 — `bb_pctile` in TOPLANIYOR `why_now`

**Before:** §23 sample card shows `BB genişliği 60g'nin alt %22'inde` as a TOPLANIYOR bullet, but the orchestrator passed `bb_pctile=None` to `why_now`. The bullet was suppressed because the percentile was never computed for the surface — the gate logic in `evaluate_toplaniyor` used the percentile cuts (`bb_width_60d_p25`, `bb_width_60d_p35`) directly without surfacing the rank.

**After:** New field `EngineInputs.bb_width_60d_pctile: Optional[float]` — the 60-bar percentile rank of `bb_width_today` (0=tightest, 100=widest). Computed in `build_engine_inputs`:

```python
if bb_today is not None and len(bb_60) >= 5:
    below = float((bb_60 < bb_today).sum())
    bb_60d_pctile = round(below / float(len(bb_60)) * 100.0, 1)
```

Threaded through to `why_now`'s TOPLANIYOR template, which already handled the value when present:

```python
toplaniyor={
    "rvol_5d_avg":  inp.rvol_5d_avg if inp else None,
    "bb_pctile":    (inp.bb_width_60d_pctile if inp is not None else None),  # ← was None
    ...
}
```

**Verification:** With a TOPLANIYOR-shaped fixture and `bb_pctile=22.0`:

```
TOPLANIYOR why_now bullets:
 • Sessiz hacimlenme (rvol 5g ort. 1.28)
 • BB genişliği 60g'nin alt %22'inde       ← now renders, was previously suppressed
 • ADX yükseliyor, trend şekilleniyor
 • Yüksek dipler — yapı sıkışıyor
```

Matches §23 sample exactly.

## What's NOT in this polish pass

The remaining limitations (#1, #2, #3, #5, #7) all need state stores, walk-forward results, or external data plumbing — they're v2 territory:

| # | Limitation | Why deferred |
|---|------------|--------------|
| 1 | Timestamp-based freshness | Needs a per-metric timestamp source the data layer doesn't currently expose. |
| 2 | `accumulation_strength` weights | Walk-forward needed to recalibrate from realized TOPLANIYOR→HIZLI/SWING upgrade rates. |

These are documented in `BULLALFA_FINAL_DELIVERABLES.md §7` and don't block v1 launch.
